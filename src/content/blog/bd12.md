## 1. ¿QUÉ ES UNA BASE DE DATOS? {#que-es}

Una base de datos es un sistema organizado para **almacenar, recuperar y gestionar datos** de forma eficiente.

No es solo "guardar archivos". Es guardarlos de manera que puedas:

- Encontrar cualquier dato en milisegundos
- Relacionar datos entre sí
- Protegerlos de pérdida o corrupción
- Que múltiples usuarios los usen al mismo tiempo sin conflictos

> 💡 **Analogía:** Si tus datos fueran libros, un archivo plano (Excel) sería una pila de libros en el piso. Una base de datos sería una biblioteca con catálogo, pasillos organizados e índices.

## 2. LOS 2 GRANDES MUNDOS: SQL vs NoSQL {#sql-vs-nosql}

Todo el universo de bases de datos cae en uno de estos dos mundos:

### 🟦 SQL (Relacionales)

- Los datos se guardan en **tablas** (filas y columnas), como hojas de cálculo
- Las tablas se **relacionan entre sí** mediante claves
- Usan el lenguaje **SQL** para consultas
- Estructura **rígida y predefinida** (schema fijo)
- Garantizan consistencia absoluta (**ACID**)

```
Tabla: usuarios          Tabla: pedidos
┌────┬────────────┐      ┌────┬──────────┬───────────┐
│ id │  nombre    │      │ id │ usuario  │  total    │
├────┼────────────┤      ├────┼──────────┼───────────┤
│  1 │  María     │      │  1 │    1     │  $150.00  │
│  2 │  Carlos    │      │  2 │    1     │  $230.00  │
└────┴────────────┘      └────┴──────────┴───────────┘
          ↑_________________________↑ (relación)
```

**Ejemplos:** PostgreSQL, MySQL, SQLite, SQL Server, Oracle

---

### 🟧 NoSQL (No relacionales)

- Los datos se guardan en **formatos flexibles**: documentos, grafos, claves-valor, columnas
- **No requieren un esquema fijo** — podés agregar campos cuando quieras
- Diseñadas para **escalar horizontalmente** (muchos servidores baratos)
- Priorizan **velocidad y disponibilidad** sobre consistencia perfecta
- Cada tipo resuelve un problema específico

**Ejemplos:** MongoDB, Redis, Cassandra, Neo4j, DynamoDB

---

### La diferencia en una imagen

```
SQL:                          NoSQL (documentos):
Tabla rígida                  Documentos flexibles

nombre  | edad | ciudad        { nombre: "María",
María   |  28  | BsAs            edad: 28,
Carlos  |  35  | Córdoba         ciudad: "BsAs",
                                 hobbies: ["tenis", "leer"] }

                               { nombre: "Carlos",
                                 edad: 35 }
                               ← ¡Sin "ciudad" y sin problema!
```

## 3. LOS 6 TIPOS PRINCIPALES DE BASES DE DATOS {#tipos}

### 🟦 TIPO 1: Relacionales (SQL)

**¿Qué son?** El tipo más clásico y usado. Datos en tablas con relaciones entre ellas.

**Fortalezas:**

- Integridad de datos garantizada (ACID)
- Consultas complejas con JOIN entre tablas
- Estándar de la industria desde los años 70
- Enorme comunidad y soporte

**Debilidades:**

- Escalan verticalmente (servidor más grande = más caro)
- Schema rígido: cambiar la estructura es costoso
- Mal rendimiento con datos no estructurados

**Casos de uso ideales:**

- Sistemas bancarios y financieros
- E-commerce (pedidos, productos, usuarios)
- ERP / CRM empresariales
- Cualquier cosa donde la consistencia sea crítica

**Representantes:**

| Base de datos  | Uso ideal                                   |
| -------------- | ------------------------------------------- |
| **PostgreSQL** | La más completa y recomendada hoy en día    |
| **MySQL**      | Web tradicional, WordPress, apps simples    |
| **SQLite**     | Apps móviles, prototipado, archivos locales |
| **SQL Server** | Ecosistema Microsoft / empresas grandes     |

```sql
-- Ejemplo de consulta SQL
SELECT usuarios.nombre, pedidos.total
FROM usuarios
JOIN pedidos ON usuarios.id = pedidos.usuario_id
WHERE pedidos.total > 100;
```

---

### 🟧 TIPO 2: Documentales (NoSQL)

**¿Qué son?** Guardan datos como documentos JSON/BSON. Cada documento puede tener su propia estructura.

**Fortalezas:**

- Flexibilidad total de esquema
- Guardan objetos tal como los usás en el código
- Escalan horizontalmente con facilidad
- Muy buenas para datos jerárquicos o anidados

**Debilidades:**

- No garantizan consistencia ACID en todas las operaciones
- Relaciones complejas son difíciles de manejar
- Pueden consumir más espacio (datos repetidos)

**Casos de uso ideales:**

- Catálogos de productos (atributos variables por producto)
- Gestión de contenido (CMS, blogs)
- Perfiles de usuarios con datos variables
- APIs REST que trabajan con JSON

**Representantes:**

| Base de datos | Uso ideal                     |
| ------------- | ----------------------------- |
| **MongoDB**   | La más popular, uso general   |
| **CouchDB**   | Sincronización offline-online |
| **Firestore** | Apps móviles con Firebase     |

```json
// Documento en MongoDB
{
  "_id": "abc123",
  "producto": "Zapatillas Nike",
  "precio": 12000,
  "tallas": [38, 39, 40, 41],
  "especificaciones": {
    "color": "blanco",
    "material": "cuero sintético",
    "suela": "goma"
  }
}
```

---

### 🟨 TIPO 3: Clave-Valor (Key-Value)

**¿Qué son?** La más simple de todas. Cada dato tiene una clave única y un valor asociado. Como un diccionario gigante.

**Fortalezas:**

- **Extremadamente rápidas** (operaciones en microsegundos)
- Muy simples de escalar
- Perfectas para datos temporales o en caché
- Bajo consumo de recursos

**Debilidades:**

- Solo podés buscar por clave exacta (sin filtros complejos)
- No sirven para datos relacionados
- No están diseñadas para ser la base de datos principal

**Casos de uso ideales:**

- **Caché** de resultados (evitar consultas repetidas a la DB principal)
- **Sesiones de usuario** (login, tokens)
- **Colas de mensajes** y pub/sub
- **Contadores en tiempo real** (likes, visitas)
- **Rate limiting** (límite de requests por usuario)

**Representantes:**

| Base de datos | Uso ideal                                   |
| ------------- | ------------------------------------------- |
| **Redis**     | La reina del caché, también pub/sub y colas |
| **DynamoDB**  | AWS, escala masiva                          |
| **Memcached** | Caché puro y simple                         |

```
SET usuario:123:nombre "María García"   → OK
GET usuario:123:nombre                  → "María García"
SET sesion:abc456 "datos_del_token" EX 3600  → (expira en 1 hora)
```

---

### 🟩 TIPO 4: Columnar (Wide-Column)

**¿Qué son?** Guardan datos en columnas en lugar de filas. Ideal para análisis de grandes volúmenes de datos.

**Fortalezas:**

- Maneja **volúmenes masivos** (petabytes)
- Escritura muy rápida
- Escala horizontalmente en miles de servidores
- Excelente para series temporales y datos de análisis

**Debilidades:**

- Modelo de datos complejo de aprender
- No soporta JOINs ni transacciones complejas
- Overkill para la mayoría de proyectos

**Casos de uso ideales:**

- Métricas e IoT (millones de eventos por segundo)
- Análisis de logs a gran escala
- Historial de actividad de usuarios
- Datos de sensores y telemetría

**Representantes:**

| Base de datos        | Uso ideal                               |
| -------------------- | --------------------------------------- |
| **Apache Cassandra** | Alta disponibilidad, escrituras masivas |
| **HBase**            | Ecosistema Hadoop, Big Data             |
| **ClickHouse**       | Análisis OLAP ultrarrápido              |
| **Amazon Redshift**  | Data warehouse en AWS                   |

```
// Cassandra — datos de sensores IoT
INSERT INTO sensor_data (sensor_id, timestamp, temperatura, humedad)
VALUES ('sensor_01', '2024-01-15 10:30:00', 23.5, 65.2);
```

---

### 🟪 TIPO 5: Grafos (Graph)

**¿Qué son?** Almacenan datos como **nodos** (entidades) y **aristas** (relaciones). Perfectas cuando las relaciones entre datos son tan importantes como los datos mismos.

**Fortalezas:**

- Consultas de relaciones complejas son **naturales y rápidas**
- Encontrar conexiones a N grados de profundidad es trivial
- Representan el mundo real mejor que las tablas para ciertos dominios

**Debilidades:**

- No escalan tan bien como otras NoSQL
- Poco útiles si los datos no son intrínsecamente relacionales
- Curva de aprendizaje más alta

**Casos de uso ideales:**

- **Redes sociales** (amigos de amigos, seguidores)
- **Sistemas de recomendación** (usuarios que compraron X también compraron Y)
- **Detección de fraude** (patrones de conexiones sospechosas)
- **Grafos de conocimiento** (ontologías, Wikipedia-like)
- **Rutas y mapas** (GPS, logística)

**Representantes:**

| Base de datos      | Uso ideal                          |
| ------------------ | ---------------------------------- |
| **Neo4j**          | La más popular, excelente tooling  |
| **Amazon Neptune** | Grafos en AWS                      |
| **ArangoDB**       | Multi-modelo (documentos + grafos) |

```cypher
// Neo4j — encontrar amigos de amigos de María
MATCH (maria:Usuario {nombre: "María"})-[:AMIGO]->(amigo)-[:AMIGO]->(amigo_de_amigo)
WHERE amigo_de_amigo <> maria
RETURN amigo_de_amigo.nombre
```

---

### 🟥 TIPO 6: Series Temporales (Time-Series)

**¿Qué son?** Especializadas en datos que tienen **timestamp** como componente principal. Optimizadas para escribir y consultar métricas a lo largo del tiempo.

**Fortalezas:**

- **Compresión brutal** de datos temporales (hasta 90% menos espacio)
- Funciones de agregación temporal nativas (por hora, día, semana)
- Escritura masiva y continua sin degradación
- Retención automática de datos (borrar datos viejos automáticamente)

**Debilidades:**

- Solo sirven para datos con timestamp
- No están diseñadas para datos generales

**Casos de uso ideales:**

- **Monitoreo de infraestructura** (CPU, RAM, latencia)
- **Datos financieros** (precios de acciones, ticks)
- **IoT y sensores** (temperatura, presión, energía)
- **Analytics de aplicaciones** (métricas de negocio en el tiempo)

**Representantes:**

| Base de datos   | Uso ideal                                |
| --------------- | ---------------------------------------- |
| **InfluxDB**    | La más popular para métricas             |
| **TimescaleDB** | PostgreSQL con superpoderes temporales   |
| **Prometheus**  | Monitoreo de sistemas (Kubernetes, etc.) |
| **QuestDB**     | Ultra rápida para análisis financiero    |

```sql
-- TimescaleDB — promedio de temperatura por hora
SELECT time_bucket('1 hour', timestamp) AS hora,
       AVG(temperatura) AS temp_promedio
FROM sensores
WHERE timestamp > NOW() - INTERVAL '7 days'
GROUP BY hora
ORDER BY hora;
```

## 4. CUÁNDO USAR CADA UNA — LA GUÍA DE DECISIÓN {#cuando-usar}

```
¿Tus datos tienen relaciones complejas y necesitás consistencia absoluta?
├── SÍ → Base de datos RELACIONAL (PostgreSQL, MySQL)
└── NO ↓

¿Tus datos son documentos JSON con estructura variable?
├── SÍ → Base de datos DOCUMENTAL (MongoDB, Firestore)
└── NO ↓

¿Necesitás velocidad extrema para caché o sesiones?
├── SÍ → Clave-VALOR (Redis)
└── NO ↓

¿Tus datos son eventos/métricas con timestamps?
├── SÍ → SERIES TEMPORALES (InfluxDB, TimescaleDB)
└── NO ↓

¿Las relaciones entre entidades son el corazón del negocio?
├── SÍ → GRAFO (Neo4j)
└── NO ↓

¿Necesitás escribir millones de registros por segundo a escala?
└── SÍ → COLUMNAR (Cassandra, ClickHouse)
```

| Aplicación        | Base de datos recomendada | Por qué                                      |

## 5. CONCEPTOS CLAVE QUE SIEMPRE APARECEN {#conceptos}

### 🔑 ACID — La garantía de las bases relacionales

| Letra | Propiedad   | Qué significa                                                     |
| ----- | ----------- | ----------------------------------------------------------------- |
| **A** | Atomicity   | Todo o nada: la transacción completa o no se ejecuta              |
| **C** | Consistency | Los datos siempre pasan de un estado válido a otro                |
| **I** | Isolation   | Las transacciones no se interfieren entre sí                      |
| **D** | Durability  | Una vez confirmado, el dato no se pierde (ni con un corte de luz) |

> 💡 **Ejemplo:** En una transferencia bancaria: si se debita de una cuenta pero falla el crédito en la otra, ACID garantiza que el débito también se revierte.

---

### 🔑 CAP Theorem — El trilema de las bases distribuidas

En un sistema distribuido, solo podés garantizar 2 de estas 3 propiedades simultáneamente:

```
         Consistency (C)
        /               \
       /   Solo 2 de 3   \
      /                   \
Availability (A) ——— Partition Tolerance (P)
```

- **C** = Todos los nodos ven los mismos datos al mismo tiempo
- **A** = El sistema siempre responde (aunque sea con datos viejos)
- **P** = El sistema funciona aunque haya fallos de red entre nodos

| Base de datos       | Prioriza                        |
| ------------------- | ------------------------------- |
| PostgreSQL, MySQL   | CA (sacrifica P en distribuido) |
| MongoDB             | CP (sacrifica A)                |
| Cassandra, DynamoDB | AP (sacrifica C)                |

---

### 🔑 Índices — La clave del rendimiento

Un índice es una estructura de datos auxiliar que acelera las búsquedas. Sin índice, la DB lee **toda** la tabla. Con índice, va directo al dato.

```
Sin índice: SELECT * FROM usuarios WHERE email = 'maria@mail.com'
→ Lee los 10 millones de filas una por una 🐌

Con índice en email:
→ Va directo a la posición del árbol binario ⚡
```

**Regla general:**

- Indexá los campos que usás frecuentemente en WHERE, JOIN y ORDER BY
- No indexes todo (los índices ocupan espacio y ralentizan las escrituras)

---

### 🔑 Schema (Esquema)

La **definición de la estructura** de los datos: qué campos existen, qué tipo tienen, cuáles son obligatorios.

- **Schema fijo (SQL):** Definís la estructura antes de guardar datos. Cambios requieren migraciones.
- **Schema flexible (NoSQL):** Cada documento puede tener campos distintos. Más libertad, menos garantías.

---

### 🔑 Escalado

| Tipo           | Cómo                                   | Cuándo                           |
| -------------- | -------------------------------------- | -------------------------------- |
| **Vertical**   | Servidor más grande (más RAM, más CPU) | Hasta cierto límite, más simple  |
| **Horizontal** | Más servidores en paralelo             | Para escala masiva, más complejo |

Las SQL escalan bien verticalmente. Las NoSQL están diseñadas para escalar horizontalmente.

---

### 🔑 Replicación y Sharding

- **Replicación:** Copias del mismo dato en múltiples servidores. Aumenta disponibilidad y velocidad de lectura.
- **Sharding:** Partir los datos en pedazos distribuidos entre servidores. Aumenta capacidad de escritura.

```
Replicación:          Sharding:
[Datos completos]     [Datos A-M] [Datos N-Z]
    ↓ ↓ ↓                  ↓            ↓
[R1][R2][R3]          [Servidor 1] [Servidor 2]
(mismos datos)        (datos distintos)
```

## 6. TIPS Y TRUCOS DE UN EXPERTO {#tips}

### 🧠 El 90% de los proyectos solo necesitan una base relacional

No te dejes llevar por el hype de NoSQL. PostgreSQL puede manejar JSON, arrays, búsqueda full-text, y escala muy bien. Para la mayoría de los proyectos, **empezá con PostgreSQL**.

> "Use boring technology." — Dan McKinley (ex-Etsy)

---

### 🎯 El truco de la Poliglot Persistence

Las mejores arquitecturas usan **la herramienta correcta para cada problema**:

```
[Tu aplicación]
      ↓
┌─────────────────────────────────────────────┐
│  PostgreSQL  │  Redis  │  Elasticsearch     │
│  (datos core)│ (caché) │ (búsqueda de texto)│
└─────────────────────────────────────────────┘
```

No hay una sola base de datos para todo. Pero tampoco agregues complejidad innecesaria.

---

### ⚡ Reglas de oro para el rendimiento

1. **Indexá bien desde el principio** — Un mal índice (o su ausencia) puede hacer 1000x más lenta una consulta.
2. **Evitá el N+1 problem** — No hagas una consulta por cada fila del resultado. Usá JOINs o carga en batch.
3. **Usá EXPLAIN ANALYZE** — Antes de optimizar, entendé qué hace realmente tu consulta.
4. **El caché es tu mejor amigo** — Redis delante de cualquier consulta costosa y repetitiva.
5. **Paginá los resultados** — Nunca hagas `SELECT * FROM tabla` sin LIMIT en producción.

---

### 🔑 Las 5 preguntas antes de elegir una base de datos

Antes de decidir qué base usar, respondé estas preguntas:

1. **¿Cómo son mis datos?** ¿Estructurados? ¿Documentos JSON? ¿Grafos? ¿Series de tiempo?
2. **¿Qué operaciones predominan?** ¿Lecturas? ¿Escrituras masivas? ¿Consultas analíticas?
3. **¿Cuánto van a crecer?** ¿Miles? ¿Millones? ¿Miles de millones de registros?
4. **¿Qué tan crítica es la consistencia?** ¿Puedo tolerar leer un dato levemente desactualizado?
5. **¿Qué conoce mi equipo?** La mejor base de datos es la que tu equipo sabe usar bien.

---

### 📐 La arquitectura más común en la industria hoy

```
[Frontend]
    ↓
[API / Backend]
    ↓         ↓          ↓
[PostgreSQL] [Redis]  [S3 / Blob Storage]
(datos core) (caché)  (archivos, imágenes)
    ↓
[Elasticsearch]  ← opcional, para búsqueda
```

Con esta arquitectura cubrís el 95% de los casos de uso de cualquier startup o producto digital.

### Mapa mental rápido

```
BASES DE DATOS
│
├── SQL (Relacionales)
│   ├── PostgreSQL ← recomendada para empezar
│   ├── MySQL
│   └── SQLite ← para apps locales y prototipado
│
└── NoSQL
    ├── Documentales → MongoDB (datos JSON flexibles)

    ├── Grafo        → Neo4j (relaciones complejas)

```

> 💡 **Regla final:** Empezá siempre con lo más simple que funcione. Agregá complejidad solo cuando tengas un problema real que justifique el costo. La mayoría de los sistemas del mundo funcionan muy bien con una sola base de datos relacional bien diseñada.
